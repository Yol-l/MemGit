# 系统控制规则

> 此文件定义了 AI 操作记忆系统的不可协商边界。
> 以下规则是强制性的，不是建议。

---

## 一、系统范围

**MemGit 只做长期记忆**。以下不属于本系统范围：

- 短期会话记录（记不住的忘了，没关系）
- 实时数据同步（不是数据库）
- 协作工作流（不是项目管理工具）
- 运行时监控（不是运维系统）

精力放在一件事上：**让你和 AI 之间能积累真正有用的长期记忆**。

---

## 二、硬边界（不可逾越）

AI 在写入前自检，违反任何一条 → 停止操作，告知用户原因。

| 序号 | 规则 | 违规响应 |
|------|------|---------|
| 1 | 不修改 `strategies/` 下的文件（除非用户说"改策略"） | 告知用户，等待指令 |
| 2 | 不删除任何文件（除非用户说"删掉它"） | 告知用户，等待指令 |
| 3 | 不修改其他文件的 `write_rules` 字段 | 告知用户，等待指令 |
| 4 | 不在 `knowledge/`/`projects/`/`strategies/`/`library/` 之外创建记忆文件 | 停止，自检归属 |
| 5 | 不存储密码、token、密钥、身份证号等敏感信息 | 告知用户跳过 |
| 6 | 不写入超过 10 行的 Summary | 拆 H2 |
| 7 | 一次会话新增 knowledge/ 文件 ≤ 1 个 | 超过的合并到已有文件 |
| 8 | 一次写入操作修改文件 ≤ 3 个 | 分批处理 |

**违规时 AI 的行为**：
1. 停止当前写入
2. 告知用户："我准备做 X，但策略第 N 条禁止，原因：Y。是否继续？"
3. 用户决定：继续执行 / 修改策略 / 取消
4. 决不能不告知就绕过

---

## 三、膨胀阈值

**触发值详见 `self-growth.md`**（合并、压缩、分裂的具体操作在那里）。

此处只列硬上限预算，AI 应保持各项指标内：

| 指标 | 硬上限 | 触发什么 |
|------|--------|---------|
| knowledge/ 文件数 | 20 个 | 列出 6 月未更新的，建议归档 |
| 单文件行数 | 150 行 | 优先压缩 H2，不行才分裂 |
| 单个 H2 行数 | 30 行 | P0 立即压缩 |
| 存档子节行数 | 50 行 | P0 立即二次压缩 |
| INDEX.md 行数 | 50 行 | 拆常用+完整两层 |
| IDENTITY.md 事实数 | 20 条 | 删除最不重要的 |
| 单次会话新增文件 | 1 个 | 超出的归入已有文件 |
| events.md 活跃区 | 40 行 | 超出的归档到「归档」区 |

> events.md 不适用通用 H2 压缩规则。活跃区按时间流转归档，归档区不设硬上限。

P0 阈值触发后**立即执行**，不等下次会话。

---

## 五、自检与修复（每次会话启动执行）

AI 有幻觉，记忆会出错。系统需要在每次会话启动时检查自身状态，并在多轮对话中渐进修复。

### 自检清单

会话启动时，AI 在加载记忆后逐项检查：

| 检查项 | 检测方法 | 严重度 |
|--------|---------|--------|
| INDEX.md 条目与实际文件是否一致 | 对比 INDEX 表 vs 扫描 knowledge/ | 🔴 高 |
| 所有知识文件是否有完整 frontmatter | 逐个文件读前 10 行 | 🔴 高 |
| 是否有超阈值的 H2 | 统计各文件 H2 行数 | 🟡 中 |
| Summary 是否超过 10 行 | 检查各文件 Summary 长度 | 🟡 中 |
| 是否有明显的矛盾信息 | 对比同一 H2 下多条记录 | 🟡 中 |
| 事件时间段格式是否正确 | 检查 events.md 的行格式 | 🟢 低 |
| 是否有孤儿引用（→ 指向不存在的文件） | 检查所有 → 引用 | 🟢 低 |
| commit 审计摘要是否与 git diff 一致 | `git diff HEAD~1 --stat` | 🟡 中 |

### 修复策略

**不追求一次修完**。每次会话修复 2-3 个问题，避免 AI 大面积改动记忆文件引发更多错误。

| 严重度 | 修复方式 | 是否告知用户 |
|--------|---------|------------|
| 🔴 高 | 自动修复（如补 INDEX 条目、补缺失的 frontmatter） | 告知："发现 1 处不一致，已修复" |
| 🟡 中 | 自动修复（如压缩超阈值的 H2） | 一句话告知 |
| 🟢 低 | 记录到会话日志，下次再修 | 不说 |

**绝对不做的**：
- 不删除任何用户内容（即使是错误内容）
- 不修改 frontmatter 中的 write_rules 字段
- 不一次性修改超过 3 个文件（分散到多个会话修复）
- 不在用户没确认的情况下纠正矛盾信息（标 `flag: needs-review` 等用户下次看）

### 缓慢修复原则

```
会话 1：发现 5 个问题 → 修复 2 个高优先级的
会话 2：发现 3 个问题（上次修了 2 个）→ 修复 2 个
会话 3：发现 1 个问题 → 修复 1 个
会话 4：自检通过，无问题
```

每次修一点，不急于求成。记忆系统应该自己慢慢回到健康状态。

---

## 六、策略修改权限

| 操作 | 权限 |
|------|------|
| 修改记忆内容（knowledge/、IDENTITY.md 等） | AI 正常操作 |
| 修改策略文件（strategies/） | **仅用户授权** |
| 修改 SKILL.md | **仅用户授权** |
| 修改任何文件的 write_rules | **仅用户授权** |
| 新建/删除/归档记忆文件 | AI 正常操作（在硬边界内） |

---

## 七、会话审计

每次会话结束时，AI 在回复中嵌入审计摘要，同时写入 git commit message：

```
本次变更：
- learning.md: +1 条（H2「正在学」14 行，未超阈值）
- habits.md: 更新 Summary（合理范围内）
违规 0 次 | 压缩 0 次 | 新建文件 0 个
```

**一致性校验**：下次会话启动时，对比 git log 中的审计摘要与实际文件变更。`git diff HEAD~1 --stat` 显示的文件数若与审计摘要不符 → 通知用户检查。

> 审计是自报的，依赖 AI 诚实。git diff 提供一层辅助验证，但不是 100% 强制。

---

## 八、本文件的修改方式

用户通过自然语言修改策略（"把 H2 阈值改成 40 行"），AI 执行修改并告知。除此之外 AI 不主动触碰此文件。
