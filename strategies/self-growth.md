# MemGit 记忆自成长与合并规则

> 需要写入、更新或压缩时加载。此处定义的阈值是唯一权威值（system-control.md 只引用，不重复）。

## 项目进度事件生命周期

**事件定义**：有持续性的行动或计划，用时间段表示。不是时间点，不是单次查询。

### 写入规则

| 规则 | 说明 |
|------|------|
| 信心标记 | 每条条目标注 `explicit`（亲口说）/ `inferred`（推断）/ `stale`（1年未确认）/ `uncertain`（可能） |
| 持续性门槛 | 学减肥、做项目、规划旅行是事件。查天气、问代码不是 |
| 时间段格式 | `(YYYY-MM-DD – YYYY-MM-DD)`，仍在进行中写 `至今` |
| 类型标记 | `[项目]` `[学习]` `[生活]` 等，AI 判断 |
| 关联文件 | 每条事件指向一个具体的 knowledge/ 或 projects/ 文件 |
| 合并规则 | 同一件事的不同阶段合并为一条，更新时间段 |

### 生命周期

```
写入 → 活跃期（近 6 月）→ 移入「进度·归档」
                                ↓
                         重新谈到该事件
                                ↓
                         从归档区取出放回活跃区
```

- 进度条目超 6 月未更新 → 移入 `## 进度·归档`，不删除

- 进度条目超 6 月未更新 → 移入 `## 进度·归档`，不删除
- 用户再次聊到该事件 → 从归档移回活跃区
- 归档区内条目按年份分组，超 3 年的压缩为一行摘要

## 合并规则：如何更新长期记忆

新增信息时，不是简单追加。先判断与已有内容的关系，选择对应操作：

### 1. 追加合并（Append）

**场景**：新信息属于已有 H2，且与旧条目不冲突。
**操作**：直接在 H2 下加新条目，更新 Summary。

```
## 当前持仓（合并前）
- 沪深300 ETF: 40%

## 当前持仓（合并后）
- 沪深300 ETF: 40%
- 比特币: 5%     ← 新增
```

### 2. 更新合并（Update）

**场景**：新信息直接取代旧信息（偏好变了、持仓变了、目标进度更新了）。
**操作**：旧条目**立即移入存档区**，活跃区只保留当前值。不要把 superseded 条目留在活跃区。

```
## 编程偏好（合并前）
- IDE: VS Code

## 编程偏好（合并后）
- IDE: Cursor (2026-07-14)

## 编程偏好·存档
- (2026-01) IDE: VS Code
```

**关键区别**：superseded 条目不在活跃区堆积——直接进存档。活跃区永远只有当前值。

### 3. 纠正合并（Correct）

**场景**：用户说"上次说的不对，其实是……"。
**操作**：直接修正原条目，添加注释说明。

```
## 当前持仓
- 腾讯: 15%     <!-- 2026-07-13 修正：之前误记为 10% -->
```

### 4. 进度合并（Progress）

**场景**：目标或学习有了新进展。
**操作**：在原条目旁标注当前状态。

```
## 年度目标
- [进行中] 跑一次马拉松 → 已完成 ✅   ← 状态更新
- [已完成] 读完 12 本书（读到第 8 本了）
```

### 5. 归并合并（Consolidate）

**场景**：用户在不同时间用不同说法表达了同一个偏好/事实。
**操作**：合并为一条，保留最早时间和最近更新时间。
**约束**：只有当两条目**语义完全相同**时才归并。只是"相关"不归并。

```
## 做事风格（合并前）
- (2026-01) 喜欢简洁的代码
- (2026-06) 讨厌过度设计  ← 语义不同，不归并

## 做事风格（合并后，不归并的情况）
- (2026-01) 喜欢简洁的代码
- (2026-06) 讨厌过度设计

## 做事风格（合并前，另一个例子）
- (2026-01) 喜欢简洁的代码
- (2026-06) 代码要简洁  ← 语义相同，归并

## 做事风格（合并后）
- 追求简洁的代码（最早 2026-01，最近确认 2026-06）
```

## 压缩规则

**触发**：某个 H2 小节超过 **30 行**。

**做法**：活跃信息保持在顶层，旧条目归入「存档」子节。

```
## 兴趣爱好（40 行，太长了）

## 兴趣爱好
- 目前活跃：攀岩、摄影、烹饪（2026）
- 过去：打游戏、跑步（<2025）

## 兴趣爱好·存档
- 2024: 打游戏（退坑），跑步（偶尔）
- 2023: 摄影入门，买了索尼 A7
```

**其他触发条件**：
- 同一话题超过 3 个时间段的记录 → 按年份分组
- 某条记录超过 6 个月无更新 → 移入存档
- 用户明确表达了信息变化 → 旧标记 `superseded`

## 冲突处理

旧信息不删，标记为过时：

```markdown
## 当前持仓（2026-07）
- 沪深300 ETF: 40%
- 腾讯: 15%

<!-- superseded 2026-03 -->
~~- 茅台: 10%~~
```

## 分裂规则

文件超过 **150 行** 时：**优先压缩，不分裂**。更多文件 = 更多检索开销，压缩比分裂好。

## 存档区二次压缩

存档子节（`## XXX·存档`）超过 **50 行** 时，把更旧的条目压缩为一行摘要：

```
## 兴趣爱好·存档（压缩前，60 行）
- 2024: 打游戏（退坑），跑步（偶尔）
- 2023: 摄影入门，买了索尼 A7
- 2022: 学吉他（放弃了），养猫
- 2021: ...

## 兴趣爱好·存档（压缩后）
- 2024: 打游戏（退坑），跑步（偶尔）
- 2023: 摄影入门，买了索尼 A7
- <2022: 吉他（放弃）、养猫等早期尝试（已归档摘要）
```

**原则**：越旧的信息压缩比越高。最近 2 年保留细节，更早的只留一行摘要。

## Summary 同步

每次修改 H2 内容后，**检查 `## Summary` 是否需要更新**。Summary 过时了，渐进式加载就白费了。

## INDEX.md 膨胀控制

INDEX.md 超过 **30 行** 时，拆分为两部分：

```
## 常用文件（最近活跃 5 个）
| 文件 | Summary |
|------|---------|
| investments.md | ... |   ← 最近更新过的
| learning.md | ... |

## 全部文件
| 文件 | Summary |
|------|---------|
| principles.md | ... |
| habits.md | ... |
| ...全部文件... |
```

AI 先读「常用文件」部分（5 行），80% 场景够用。需要时再读「全部文件」部分。
