为什么 AI 需要记忆 — 核心痛点

大语言模型本质上是无状态的——每次推理都是独立事件。你上周告诉它的偏好、你们共同做的决策，在下一次对话中全部归零。

痛点 1

上下文窗口有限

最先进的模型约 200K token（约 15 万字中文）。对话一长，早期信息被挤出窗口。像只能记住最近 20 分钟的人。

痛点 2

AI 有幻觉

AI 可能"记住"从未发生的事，并自信地复述。用户很难察觉，错误信息持续污染后续交互。

痛点 3

跨会话空白

新对话从零开始。一周后回来，AI 连你叫什么都不知道。没有连续性 = 没有真正的关系。

本质问题：上下文窗口 ≠ 记忆。窗口是"短期工作台"，记忆是"跨会话持久化"。解决的不是"装更多"，而是"跨时间保留"。

记忆的三条技术路线 — 实现方式

路线一 · 最成熟

外置记忆（非参数化）

原理：信息存在模型外部的数据库（向量库 / 知识图谱 / KV 存储），对话时检索相关片段注入上下文。

核心：把"记忆"转化为"检索"问题。RAG 是最典型的形态。

代表：Mem0 · Letta · Zep/Graphiti · LangMem

路线二 · 效果最好

参数化记忆

原理：通过微调（SFT / LoRA）将知识直接编码进模型权重——像"肌肉记忆"，不需要检索就能用。

核心：记忆与模型融为一体。推理速度快，但更新成本高。

代表：Character.AI · 领域微调模型

路线三 · 最新

隐式记忆

原理：不存具体内容，存"推理过程中的中间状态"——KV Cache 智能管理，压缩历史对话的注意力模式。

核心：让模型"隐约记得"而不需要完整回放。

代表：Google CoMEM · KV Cache 压缩

趋势：三条路线不互斥。混合架构正在成为主流——参数化（高频技能）+ 外置（历史事实）+ 隐式（当前会话上下文压缩）。但绝大多数生产系统以外置记忆为主。

头部 AI 产品的记忆实现 — ChatGPT / Claude / Gemini

ChatGPT

双层自动记忆

显式记忆：用户或 AI 标记的事实，可在设置中查看和删除。

隐式记忆（Dreaming）：2025 年 4 月上线，后台扫描全部对话历史自动合成记忆。无需用户明确说"记住"。

问题：黑箱——用户看不到 Dreaming 如何理解你，无法审计推理过程。

Claude

透明文件记忆

Markdown 文件：记忆存为可读的 .md 文件，用户可以逐条查看、编辑、删除。

项目隔离：Projects 功能实现工作/个人记忆隔离。2026 年 3 月支持从 ChatGPT 导入。

优势：最透明可控。记忆可导出迁移，不锁定在平台内。

Gemini

生态整合记忆

Personal Context：打通 Google 全家桶——Gmail 邮件、Docs 文档、Calendar 日程、Photos 照片、搜索历史。

默认开启：2025 年 8 月起自动整合，用户需手动关闭。

风险：跨产品超级画像，隐私控制最弱。EEA/UK/瑞士被排除。

三种架构对比：ChatGPT 是自动提取式（最便利但最不透明）；Claude 是用户触发式（最透明可控）；Gemini 是生态整合式（最深入但隐私风险最高）。共同问题：数据锁定在平台内，用户无法真正拥有自己的记忆。

我感兴趣的方向 — Git + Markdown 管理记忆

不是"更好的方案"，只是我在探索的一个方向。核心想法：用户拥有自己的记忆数据，AI 来读写，但不持有。

设计思路

文件即记忆

记忆不是数据库里的向量，是你可以直接打开阅读的 Markdown 文件。

每个文件自带 frontmatter（摘要 + 读取时机 + 写入规则），AI 打开文件第一眼就知道：这是什么、什么时候读、怎么写。

用 Git 做版本控制——每次修改都有 commit 记录，改坏了能回滚。

为什么选 Git

四个理由

1. 版本控制——记忆的每次变更可追溯，回滚到任意历史版本

2. 多设备同步——push/pull 就能跨设备保持记忆一致

3. 用户拥有数据——不依赖任何 AI 平台，换工具记忆跟着你走

4. 可分享——fork 仓库就能把记忆策略分享给别人

与市面方案的区别：Mem0/Letta 把记忆存在数据库里，用户看不到也改不了。ChatGPT 的记忆是黑箱。我的方向是——记忆就是文件，放在你自己的 Git 仓库里，任何 AI 工具都能读，但你随时可以打开编辑。

系统结构 — 目录即架构

memories/ ← 你的 Git 仓库 ├── IDENTITY.md # 你是谁（每次会话先读） ├── INDEX.md # 记忆导航图（有哪些文件、什么时候读） ├── MEMORY-ARCHITECTURE.md # 架构总览（AI 理解系统的入口） ├── CHANGELOG.md # 架构演化日志 │ ├── knowledge/ # 个人长期记忆 │ ├── events.md # 近期持续性事件（时间段格式） │ ├── principles.md # 价值观、做事原则 │ ├── habits.md # 工作习惯、沟通偏好、情绪模式 │ ├── learning.md # 学习计划、技能进展 │ ├── life.md # 兴趣、健康、重要关系 │ └── goals.md # 人生目标、年度目标 │ ├── projects/ # 项目记忆（一个项目一个目录） │ ├── INDEX.md # 项目索引 │ ├── memgit/ # MemGit 项目 │ │ └── README.md # 项目决策 + 进度 │ └── archive/ # 已归档项目 │ ├── library/ # 大段代码/资料（按需读，不进上下文） │ └── strategies/ # 记忆维护策略（可共享给他人） ├── system-control.md # 硬边界 + 膨胀阈值 + 自检规则 ├── self-growth.md # 合并、压缩、冲突、分裂规则 ├── proactive-usage.md # 主动使用记忆的规则 └── emotional-awareness.md # 情绪感知规则

纯文本 · 无平台依赖 · Git 版本控制 · 每个文件 frontmatter 自带读写规则 · 任意 AI 工具通用

渐进式加载 — 文件再大也不影响 AI 效率

核心思路：AI 不是从头读到尾，而是用 H2 标题做锚点按需跳读。

第 1 层：扫 Summary

3-5 行概括

每个文件顶部有 ## Summary，3-5 行说明这个文件记录了什么。

AI 扫一眼 Summary 就知道这个文件值不值得打开——约 40 token，2 秒。

第 2 层：跳 H2

按标题定位

打开文件后不全文读，只看相关的 ## H2 小节。

比如聊投资策略 → 只读 ## 投资理念，不读 ## 交易记录。

关键点：文件从 50 行长到 5000 行，AI 的阅读成本不变——因为只读 Summary + 1 个 H2 ≈ 170 token。H2 标题就是 AI 的索引。前提：每个大类独立成 H2，不能散布全文。

我的核心关注点 — 系统的存在意义

① 自成长（不可修改）

系统自己演化架构、新增分类、自我优化。包含：渐进自检（分批检查记录进度）、养护优先（不把资源全花在自查上）、主动感知（交互越多成长越快）、膨胀合并（持续检查文件是否过大、是否有可合并内容）。

② 异常恢复（不可修改）

AI 幻觉自动纠偏、记忆损坏恢复、规则冲突自行检测。用户否认立即修正，同级记忆降级不再引用，自然话题重新确认后恢复。

③ 只记长期

6 个月后还有价值的信息才进记忆。不记流水账、不记一次性信息。

④ 规则在先

先定规则再建文件。每条写入经过 8 条硬边界自检。AI 不能自由发挥。

⑤ 主动感知

根据记忆主动给建议、关联方案，根据情绪调整话术。不是等用户问，而是积极交互。

⑥ 记忆可纠

每轮对话都是校验机会。用户否认立即修正——记忆不是事实，是待验证的断言。

案例：AI 幻觉的自我修复 — 核心能力演示

这是整个系统最核心的能力——AI 写入了错误记忆，如何在后续对话中自然修复。

用户我喜欢吃榴莲
AI记住了，你喜欢吃榴莲。
→ 写入 life.md：【喜欢吃榴莲】标记为「确认的」
用户（下次会话）最近想吃什么
AI你之前说喜欢榴莲，要不要买点？
→ 正常引用记忆，用户没否认 → 记忆有效
用户（某次会话）帮我推荐几家餐厅
AI你之前说喜欢臭豆腐，推荐这几家...
用户我什么时候说喜欢臭豆腐了？我说的是榴莲
AI抱歉，是我记错了。已修正——你喜欢的是榴莲，不是臭豆腐。
→ 立即修正错误记忆
→ 同批次写入的其他记忆降级为「存疑的」
→ 「存疑的」记忆不再主动引用，直到自然话题重新确认
核心机制：不存原始对话、不做外部验证。纯靠"用错了 → 用户否认 → 修正 + 降级 → 不再用错 → 自然确认后恢复"这个闭环。每一轮对话都是修复机会，用得越久记忆越准。

设计理念 — 四个核心判断

记忆是待验证的断言

AI 有幻觉，写入时可能出错。但不存原始对话、不做外部验证——纯靠后续对话中用户否认来修复。用得越久越准。

用户拥有数据

纯 Markdown + Git。数据在你的仓库里，不在某个 AI 平台的黑箱里。换工具记忆跟着你走。你可以随时打开编辑。

自成长 > 完美初始设计

不追求一次性设计完美。系统在每一轮对话中自我检查、修复、演化。30+ 轮模拟验证证明了这一点——每轮都在变好。

规则决定一切

AI 不能自由发挥。每个文件自带 write_rules，每次写入经过 8 条硬边界自检。规则是系统的骨架，记忆是填充。

已知局限与展望 — 诚实面对

局限 1

无检索引擎

纯文件方案没有向量检索。文件超过 50 个时，AI 扫描 INDEX.md 的成本上升。当前靠文件结构和 H2 跳读缓解，但不是长久之计。

局限 2

策略靠 AI 自觉执行

硬边界、自检规则都是文本策略，依赖 AI 诚实执行。没有外部执行器（如 pre-commit hook）做强制。这是纯文本策略系统的天花板。

局限 3

单用户设计

一个 Git 仓库 = 一个用户的记忆。没有多用户隔离、权限管理、团队协作能力。适合个人使用，不适合企业级。

展望

下一步探索

· 接入轻量检索（如 ripgrep 全文搜索）解决大规模查找

· 加 pre-commit hook 做硬约束的强制执行

· 实际使用 3 个月，用真实数据验证自成长效果
